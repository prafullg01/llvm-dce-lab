What is LLVM?

          
LLVM (Low Level Virtual Machine) is a collection of modular and reusable compiler and toolchain technologies. It provides a modern, SSA-based compilation strategy capable of supporting both static and dynamic compilation of arbitrary programming languages.

          

            Source Code
C / C++ / Rust→
            LLVM IR
Intermediate→
            Passes
Optimize→
            Machine Code
x86 / ARM
          

          
The key insight: by using a common intermediate representation (IR), LLVM can support many source languages and target architectures while sharing the same optimization passes.

        

        

          
LLVM IR — The Language of Optimization

          
LLVM IR is a typed, SSA-based intermediate representation. Every value is assigned exactly once, and every instruction produces a named result.

          
define i32 @add(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

          

            
Types
i1, i8, i32, i64, float, double, ptr

            
Values
%name = local, @name = global

            
Basic Blocks
Sequence of instructions ending with a terminator

            
Functions
define ret_type @name(params)

          

        

        

          
SSA Form — Static Single Assignment

          
In SSA form, every variable is assigned exactly once. This makes def-use chains trivial to compute — each value has exactly one definition and a list of all instructions that use it.

          
%a = add i32 %x, 1      ; %a defined exactly once
%b = mul i32 %a, 2      ; %a used here
%c = add i32 %a, %b     ; %a used again

          
PHI nodes handle values from different control flow paths:

          
%result = phi i32 [ %a, %left ], [ %b, %right ]
; Value is %a if from %left, %b if from %right

        

        

          
Dead Code Elimination

          
An instruction is dead if its result is never used AND it has no side effects.

          
bool isDeadInstruction(Instruction &I) {
    if (!I.use_empty())         return false; // has users → LIVE
    if (I.isTerminator())       return false; // control flow → LIVE
    if (I.mayHaveSideEffects()) return false; // side effects → PROTECTED
    return true;                               // → DEAD
}

          

            
✅ Safe to Remove
add, sub, mul, icmp, zext, trunc, getelementptr, select, phi (if unused)

            
❌ Never Remove
store, call, invoke, fence, atomicrmw, volatile, landingpad, terminators

          

        

        

          
Why Some Instructions Can't Be Removed

          

            
💾
store
Modifies memory visible to other code, threads, or hardware

            
📞
call / invoke
May perform I/O, throw exceptions, modify global state

            
🔒
fence
Enforces memory ordering for concurrent programs

            
⚛️
atomicrmw
Atomic operations for lock-free synchronization

            
⚡
volatile
Hardware-mapped I/O — reads/writes trigger physical effects

            
🛑
landingpad
Exception handling entry point — part of unwind table

          

        

        

          
LLVM Pass Pipeline

          
LLVM organizes optimizations as passes. This project implements two:

          

            Frontend
clang→
            Unoptimized IR→
            Analysis Pass
Find dead code→
            Transform Pass
Remove dead code→
            Backend
Machine code

    -passes="dead-code-transform" input.ll -S -o output.ll
